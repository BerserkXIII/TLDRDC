# 🤝 Cómo Contribuir a TLDRDC

Primero: **gracias por estar aquí**. En serio. 
No me puedo llegar a considerar programador, asi que seguramente no tendre manera de solucionar posibles reportes de bugs o PRs. No busco "sacar" el juego, todavia, pero si comenzar a tratarlo de manera "profesional" mientras aprendo.
Actualmente estoy aprendiendo a programar con Python, y por motivos laborales he de enfocar mis estudios en una direccion que no aporta mucho a este proyecto. Por eso, si me das algo de conocimiento extra, lo agradeceré. <3

---

## 🎯 Siendo Honesto (Leer Esto)

Soy un **programador principiante** que aprendió a hacer esto con **MUCHA ayuda de IA**. La arquitectura funciona, pero hay partes que **no controlo 100%**. 

Eso significa:
- ✅ Puedes reportar bugs y yo haré mi mejor esfuerzo
- ✅ Puedo recibir PRs si son changeamigos
- ⚠️ Cambios arquitectónicos grandes... necesito entender primero
- ❓ Si algo se nos rompe, vamos a debuggearlo juntos

**Lo que REALMENTE necesito**: No solo fixes de código, sino **ayuda para entender**. Si me contribuyes algo, explícame POR QUÉ es así. Necesito aprender.

---

## 🚀 Tipos de Ayuda que Busco (En Orden de Preferencia)

### 1️⃣ **Explicaciones y Educación** ⭐⭐⭐ (Lo más valioso)

```
Tú: "En tu events.py, podrías mejorar esto porque..."
Yo: *aprende para siempre*
```

**Ejemplos**:
- "Deberías usar `functools.cache` aquí para optimizar"
- "Esta arquitectura tiene un problema circular potencial porque..."
- "Este patrón se llama X y te serviría para Y"
- "¿Pensaste en usar decoradores aquí?"

**Cómo participar**:
- Abre una **Discussion** en GitHub (no Issue)
- O comenta en un PR con sugerencias educativas

### 2️⃣ **Bugs y Problemas** ⭐⭐ (Importante)

```
Tú: "Cuando hago X, pasa Y"
Yo: @debuggeamos juntos
```

**Reporta**:
- El juego crashea en situación X
- Un evento no funciona correctamente
- La UI se ve rota
- Performance muy lenta

[Ver template abajo]

### 3️⃣ **Código que Me Ayude a Aprender** ⭐ (Bienvenido)

```
Tú: "Refactoricé esta función, mira:"
Yo: "¡Entiendo! ¿Me explicas por qué?"
```

**Aceptaré PRs que**:
- Sean pequeños y enfocados
- Vengan con EXPLICACIÓN de cuándo/por qué
- No cambien mucho la arquitectura
- Incluyan comentarios si es código complejo

**NO aceptaré PRs que**:
- Reescriban módulos enteros
- Cambien patrones sin explicar
- Agreguen dependencias sin consenso

## 🐛 Reportar Bugs: Template Sencillo

Abre un **Issue** con esto:

```markdown
## ¿Qué pasó?
[Descripción clara]

## ¿Cómo lo reproduzco?
1. Hice...
2. Luego...
3. Y pasó...

## ¿Qué esperaba?
[Lo que debería haber pasado]

## Mi contexto
- Python: 3.X
- OS: Windows/Mac/Linux
- ¿Algún mensaje de error? [pega aquí]
```

---

## 💡 Sugerir Mejoras: Template Sencillo

Abre un **Discussion** con:

```markdown
## Idea
[Qué se te ocurrió]

## Por qué
[De verdad, ¿por qué sería útil?]

## Cómo lo haría yo
[Tu propuesta]
```

## 📝 Si Quieres Contribuir Código

### Lo Importante es Esto:

1. **Sé pequeño**: Una función, no todo el módulo
2. **Sé claro**: En el PR, explica QUÉ cambias y POR QUÉ
3. **Sé educativo**: Enséñame algo nuevo si es posible
4. **Crea un Issue/Discussion primero**: Así vemos si estamos de acuerdo

### Proceso:

1. Abre una **Discussion** o **Issue**: "Estoy pensando en hacer X"
2. Espera mi respuesta (rápidamente, prometo)
3. Si todo ok, hace tu cambio
4. Abre el PR con descripción clara

### Tipos de PR que me encantarían:

✅ **PEQUEÑOS Y CLAROS**:
- "Optimicé esta función con [patrón] porque [razón]"
- "Agregué type hints porque mejora claridad"
- "Refactoricé x funciones relacionadas porque [razón]"

❌ **GRANDES Y MISTERIOSOS**:
- "Reescribí todo events.py" (sin discutir antes)
- "Cambia la arquitectura porque sí"
- Sin explicación de cambios
